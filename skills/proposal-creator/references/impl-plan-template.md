# Implementation Plan Template — Structure Reference

This file documents the required structure for every Implementation Plan generated by the proposal-creator skill.
Model your output after `docs/ENVDB-4728-implementation-plan.md`.

---

## Header Block

```markdown
# Implementation Plan: {Full Product / Feature Name}

**Ticket**: {TICKET-ID or "TBD"}
**Date**: {YYYY-MM-DD}
**PRD**: [{PRD filename}](./{PRD filename})
**Tool location**: `{primary code directory}`
**Language / Stack**: {Primary language and version, e.g. Python 3.12, .NET 8, TypeScript}
```

Guidelines:
- Always include a relative link to the PRD in the same directory
- If no ticket ID yet, use "TBD" — leave a placeholder

---

## Overview Section

```markdown
---

## Overview

{2–4 sentence description of what is being built and how it works end-to-end.}

Work is split into **{N} milestones** and **{M} tasks**. Tasks within a milestone are mostly independent and can be parallelised; cross-milestone dependencies are explicit.
```

Guidelines:
- Keep to 2-4 sentences max
- State the milestone and task count explicitly
- Describe the flow: input → processing → output

---

## Milestone Summary Table

```markdown
---

## Milestone Summary

| Milestone | Goal                        | Tasks                     | Output                         |
| --------- | --------------------------- | ------------------------- | ------------------------------ |
| **M0**    | {Prerequisites / setup}     | T-0.1, T-0.2 – T-0.N     | {What M0 produces}             |
| **M1**    | {MVP}                       | T-1.1 – T-1.N            | {What M1 produces}             |
| **M2**    | {Hardening / production}    | T-2.1 – T-2.N            | {What M2 produces}             |
```

Guidelines:
- Always have M0, M1, M2 as the base structure (adjust names to fit domain)
- M0 = Infrastructure setup, design freeze, prerequisites
- M1 = First working version / MVP
- M2 = Reliability, error handling, observability, production-readiness
- Tasks column: list task IDs, use em-dash for ranges (T-1.1 – T-1.7)
- Output column: describe the concrete deliverable, not a vague goal

## Task Dependency Graph

```markdown
---

## Task Dependency Graph

\`\`\`
T-0.1 ──► T-0.2 ──────────────────────────────────► T-1.3
T-0.1 ──────────────────────────────────────────► T-1.2
T-0.3 ──► T-1.1 ──► T-1.2 ──► T-1.3 ──┐
                     │         T-1.4 ──┤
                     │                 ▼
                     └──────────► T-1.5 ──► T-1.6
\`\`\`
```

Guidelines:
- Use ASCII art: `──►` for dependency arrows
- Each task appears exactly once on the left side of the graph
- Read: "T-0.1 must complete before T-0.2 can start"
- Use `──┐` and `▼` for merge points (multiple tasks converging)
- Keep the graph left-to-right, time flowing rightward
- Every task mentioned in the milestone sections must appear in this graph

## File Layout (End State)

```markdown
---

## File Layout (end state)

\`\`\`
{project-root}/
  {primary-dir}/
    {file1}           ← {T-X.Y}  {brief description}
    {file2}           ← {T-X.Y}  {brief description}
    {subdir}/
      {file3}         ← {T-X.Y}  {brief description}

{output-dir}/         ← {T-X.Y}  {Generated output root}
  {pattern}/
    {file}
\`\`\`
```

Guidelines:
- Show the complete directory tree at end-of-project state (all files that will exist)
- Annotate each file with the task ID that creates it (`← T-1.1`)
- Annotate each file with a brief description of its purpose
- Include both source files and generated output directories
- Use `←` alignment for the task annotations

## Per-Milestone Sections

### Milestone Header

```markdown
---

## M0 — {Milestone Name}
```

Use `---` separator before each milestone. Milestone name examples:
- `M0 — Prerequisites`
- `M1 — MVP`
- `M2 — Hardening`

---

### Per-Task Structure

Each task follows this exact structure:

```markdown
### T-{milestone}.{number} — {Task Name}

**Owner**: {Developer / DBA / DevOps / Product}
**Depends on**: {T-X.Y, T-X.Z, or "nothing"}
**Blocks**: {T-X.Y, T-X.Z, or "nothing (last task)"}

**Goal**: {One sentence — what this task achieves and why it matters.}

**{Relevant heading — Steps / Files / API / CLI / Config}**:

{Steps, tables, code blocks, or specifications for implementing this task}

**Acceptance criteria**:

- [ ] {Specific, testable criterion 1}
- [ ] {Specific, testable criterion 2}
- [ ] {Edge case or security criterion}
- [ ] {At least 3–5 criteria per task}
```

#### Task naming conventions

- T-0.x = M0 prerequisites
- T-1.x = M1 MVP tasks
- T-2.x = M2 hardening tasks (or T-3.x if following the example pattern)
- Number tasks sequentially within each milestone

#### Owner values

Use one of: `Developer`, `DBA`, `DevOps`, `Product`, `Tech Lead`

#### Goal sentence

One sentence only. Should complete: "The goal of this task is to..."

#### Steps section heading

Choose a heading appropriate to the task type:
- `**Steps**:` — for procedural tasks (DBA setup, one-time operations)
- `**Files**:` + code block — for tasks creating new files
- `**CLI arguments**:` + table — for CLI design tasks
- `**API**:` + code block — for interface design tasks
- `**Config**:` — for configuration tasks

#### Acceptance criteria guidelines

- Use checkbox format: `- [ ]`
- Each criterion must be independently verifiable
- Include at least one "happy path" criterion
- Include at least one "negative/edge case" criterion
- Include a security-related criterion if the task touches auth, secrets, or data access
- Be specific: "returns 3 rows" not "returns results"; "`exit code 0`" not "works"
- For file-producing tasks: include a criterion that the file is valid/parseable

## Pre-push Checklist Section

```markdown
---

## Pre-push Checklist

Before marking any milestone as complete:

- [ ] All tasks in the milestone have acceptance criteria checked
- [ ] `{primary CLI command}` runs without error
- [ ] {Build/install command} completes cleanly
- [ ] No secrets ({type of secrets}) appear in any committed file
- [ ] {Key validation check — e.g. config file parses, tests pass, lint clean}
- [ ] {Output format validation — e.g. "All generated files are valid JSON/YAML/Markdown"}
```

Guidelines:
- Tailor to the project (substitute the actual CLI command, build command, etc.)
- Always include: "No secrets appear in any committed file"
- Always include at least one format/validity check for the primary output
- 5-8 items is the right length

---

## Open Questions Blocking Implementation

```markdown
---

## Open Questions Blocking Implementation

The following PRD §12 questions must be resolved before the corresponding tasks can be finalised:

| Question                              | Blocks       | PRD Ref  |
| ------------------------------------- | ------------ | -------- |
| {Question description}                | T-X.Y, T-X.Z | §12 Q{N} |
| {Question description}                | T-X.Y        | §12 Q{N} |
```

Guidelines:
- Only include questions that actually block a task from starting or completing
- Link to the PRD section and question number (§12 Q3, §12 Q5, etc.)
- If all PRD questions are resolved (all ✅), write: "No open questions — all PRD questions resolved."
- Each row maps to an _(Open)_ item in PRD §12

## Implementation Plan Formatting Checklist

Before finalising the implementation plan, verify:

- [ ] Header block present with ticket, date, PRD link, tool location, language
- [ ] Overview section states milestone and task count
- [ ] Milestone Summary table has M0, M1, M2 rows with specific outputs
- [ ] Task Dependency Graph is an ASCII diagram with all tasks present
- [ ] File Layout shows end-state tree with task annotations
- [ ] Every task has: Owner, Depends on, Blocks, Goal, at least one Steps/Files section
- [ ] Every task has at least 3 acceptance criteria checkboxes
- [ ] Pre-push Checklist is present with 5-8 items
- [ ] Open Questions table is present (or states "no open questions")
- [ ] All task IDs referenced in "Depends on" / "Blocks" match actual task IDs defined
- [ ] No placeholder text left (e.g. `{Task Name}`, `TBD`)

## Task Count Guidelines

| Project complexity | Suggested tasks |
|-------------------|-----------------|
| Small feature (<1 week) | 4-8 tasks, 2 milestones (M0 + M1) |
| Medium feature (1-3 weeks) | 8-14 tasks, 3 milestones (M0 + M1 + M2) |
| Large feature/tool (>3 weeks) | 14+ tasks, 3+ milestones |

Lean toward more tasks with clearer scope over fewer tasks with vague scope.
Each task should represent 0.5–2 days of work for one developer.
