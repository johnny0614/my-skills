# PRD Template — Section Reference

This file documents the required sections for every PRD generated by the proposal-creator skill.
Model your output after `docs/PRD_Automated_Database_Documentation_SQLServer.md`.

---

## Product Header Block (no section number — top of file)

```markdown
# Product Requirements Document (PRD)

**Product**: {Full Product Name}
**Primary Goal**: {One sentence: what this achieves}
**Primary Users**: {Comma-separated user groups}
**Tech Stack / Datastore**: {e.g. .NET 8, SQL Server 2017, Python 3.12, AWS Lambda}
**Tenancy**: {Single-tenant / Multi-tenant + brief note}
**Scope Mode**: {e.g. "Metadata only, no LLM, no data sampling" or "MVP feature set only"}
**Output**: {e.g. "Markdown files", "REST API endpoints", "Email notifications"}
**Auth/Access**: {e.g. "JWT + role-based", "ODBC SQL Auth read-only", "OAuth2"}
```

All fields required. Use **bold** for emphasis on constraints. Add extra fields if relevant (e.g. **Environments**, **Compliance**).

## § 1 — Problem Statement & Objectives

```markdown
## 1) Problem Statement & Objectives

- **Problem**: {What is broken, painful, or missing today?}
- **Objective**: {What will this product/feature achieve?}
- **Outcomes**
  - {Measurable outcome 1}
  - {Measurable outcome 2}
  - {Measurable outcome 3}
```

Guidelines:
- Keep problem statement to 1-2 sentences
- Outcomes should be measurable or observable (not vague)
- Avoid solution language in the problem statement

## § 2 — In Scope / Out of Scope

```markdown
## 2) In Scope / Out of Scope

### In Scope (v1)

- {Feature/capability 1}
- {Feature/capability 2}
- {Feature/capability 3 — include technical specifics where relevant}

### Out of Scope (v1)

- {What is explicitly NOT being built}
- {Future phase items deferred}
- {Technical approaches ruled out}

> **Note**: {Any important clarifying note about scope boundaries}
```

Guidelines:
- Be specific — "email notifications" is better than "notifications"
- Out of scope should include why (e.g. "deferred to v2", "confirmed not applicable", "web UI — CLI only")
- Add a `> **Note**` callout for any scope decision that needed explicit discussion

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| {name} | {type} | Yes/No | {description} |

## § 5 — Non-Functional Requirements

```markdown
## 5) Non-Functional Requirements (Proposed for v1)

- **Scale**: {Expected data volumes, user counts, request rates}
- **Performance**: {Latency targets, throughput requirements}
- **Reliability**: {Uptime, retry behaviour, data consistency}
- **Security**: {Encryption, auth, secrets management}
- **Observability**: {Logging, metrics, alerting}
- **Availability**: {SLA or "CLI only / no uptime requirement"}
- **Data Residency**: {Where data lives, egress restrictions}
```

Guidelines:
- Use "Proposed for v1" in the heading — signals these are targets, not commitments
- Include units: "≤ 10 minutes", "99.9% uptime", "≤ 500ms p99"
- If a category is not applicable, say "N/A" with a brief reason

## § 6 — System Design

```markdown
## 6) System Design

### 6.1 Architecture (Runtime)

- {High-level architecture — how components connect}
- Step-by-step flow:
  - Step 1: {Input / trigger}
  - Step 2: {Processing}
  - Step 3: {Output / side effects}

### 6.2 {Key Component 1}

{Description of the most important technical component}

Use tables for component-to-responsibility mapping:

| Component | Responsibility |
|-----------|----------------|
| {name}    | {what it does} |

Use code blocks for key interfaces or data contracts.

### 6.3 {Key Component 2}

{Description, with code examples if helpful}
```

Guidelines:
- Lead with the runtime architecture (how it runs, not how it's organised)
- Include a step-by-step numbered flow for the main path
- Use tables for component inventories
- Bold DBA prerequisites or one-time setup steps

## § 7 — Security & Compliance

```markdown
## 7) Security & Compliance

- **Read/Write access**: {What data is accessed and at what privilege level}
- **Encryption in transit**: {TLS version, cipher requirements}
- **Secrets**: {How credentials are managed — never hardcoded, never logged}
- **Auditability**: {What is logged, how runs are traceable}
- **Compliance**: {GDPR, SOC2, PII handling, or "N/A — internal tooling only"}
```

Guidelines:
- Always address secrets management explicitly
- If the product handles PII, describe how it is protected
- For internal tools with no external exposure: "N/A — internal tooling" is acceptable

## § 8 — UX & Content Examples

```markdown
## 8) UX & Content Examples

### {Example 1 — most important output or interaction}

{Brief description of what the example shows}

\`\`\`{language or "markdown"}
{Concrete example of the output, UI, API response, email, etc.}
\`\`\`

### {Example 2}

\`\`\`
{Example}
\`\`\`
```

Guidelines:
- Show at least 2 concrete examples of what the product produces
- Use realistic-looking example data (not `foo`/`bar`)
- For CLIs: show the command and its output
- For APIs: show request + response
- For UIs: ASCII wireframe or describe key components
- For emails/notifications: show the full message template

## § 9 — Telemetry, Errors, and Validation

```markdown
## 9) Telemetry, Errors, and Validation

- **Logs**: {Format — structured JSON / plain text; what is logged per operation}
- **Exit codes**: {Non-zero conditions; what each exit code means}
- **Validation**:
  - {Check 1 — what invariant is verified}
  - {Check 2}
  - {Check 3}
- **Metrics**: {What is measured — counts, durations, error rates}
```

Guidelines:
- Structured logging (JSON) is preferred for CI/CD tools
- Define exit codes explicitly for CLI tools
- Validation checks should be invariants the system must never violate

## § 10 — Release Plan & Milestones

```markdown
## 10) Release Plan & Milestones (Proposed)

- **M0 ({Milestone name})**: {Goal} (≤ {N} week(s))
- **M1 ({Milestone name})**: {Goal} (≤ {N} weeks)
- **M2 ({Milestone name})**: {Goal} (≤ {N} week(s))
```

Guidelines:
- M0 = prerequisites and design freeze
- M1 = MVP / first working version
- M2 = hardening, reliability, production-readiness
- Use "≤ N weeks" for estimates — not specific dates
- Add M3 if the scope warrants a third phase

## § 11 — Acceptance Criteria & KPIs

```markdown
## 11) Acceptance Criteria & KPIs (Proposed)

- **Coverage**
  - {Coverage metric 1 — e.g. "≥ 99% of X appear in output"}
  - {Coverage metric 2}
- **Quality**
  - {Quality metric — e.g. "0 failures due to Y across 2 consecutive weeks"}
- **Performance**
  - {Performance target — e.g. "≤ 10 minutes per 1,000 records"}
- **Security**
  - {Security invariant — e.g. "No raw data values ever extracted; secrets never logged"}
```

Guidelines:
- Make KPIs measurable (percentages, counts, durations)
- Include at least one security invariant
- "Proposed" in the heading signals these are targets for team review

---

## § 12 — Open Questions for Finalization

```markdown
## 12) Open Questions for Finalization

> **Resolved questions** are marked ✅. Remaining questions require team decisions before implementation.

1. **{Question topic}** ✅ **Resolved → {Decision}**: {Brief explanation of the decision made}
2. **{Question topic}** _(Open)_: {Description of what needs to be decided}
3. **{Question topic}** _(Open)_: {Description}
```

Guidelines:
- Start numbering from 1
- Mark resolved questions with ✅ and the decision made
- Unresolved questions get _(Open)_ tag
- Open questions should be things that will block implementation if not answered
- Convert these directly to the "Open Questions Blocking Implementation" table in the implementation plan

## PRD Formatting Checklist

Before finalising the PRD, verify:

- [ ] Product header block present with all required fields
- [ ] All 12 core sections present (§1–§12)
- [ ] Section headings match exactly: `## 1) Problem Statement & Objectives`, etc.
- [ ] In scope / out of scope sections are specific (no vague entries)
- [ ] At least 2 code block examples in §8
- [ ] Milestones in §10 use "≤ N weeks" format
- [ ] KPIs in §11 are measurable
- [ ] Open questions in §12 are numbered and tagged (✅ or _(Open)_)
- [ ] No placeholder text left (e.g. `{ProductName}`, `TBD`)
